# リズムダンジョン(仮) 設計書

日本一ソフトウェア『マッドラットデッド』型の **リズムアクション × 自動生成ダンジョン**。
お気に入りの楽曲に合わせて拍タイミングでボタンを押すとキャラクターが行動し、
自動生成された横視点ダンジョンを踏破してクリアを目指す。

## 0. 決定事項

| 項目 | 決定 |
|---|---|
| 視点・移動方式 | 横視点プラットフォーマー(見た目はリアルタイム、内部は拍ゲート式) |
| 対象デバイス | スマホのみ(横画面前提)。タッチUIに全振り |
| 音源の置き方 | リポジトリに直接コミット(`songs/`)。曲ごとに BPM/offset/譜面データを持つ |
| 技術構成 | 素の HTML/CSS/JS + Canvas。ライブラリ・ビルドシステム・npm依存なし |
| 役割分担 | 設計・レビュー = メインセッション(Fable 5) / 実装 = implementer-opus(難所)・implementer-sonnet(定型) |
| セーブ | localStorage(コイン・装備・キャラ解放・章進行・較正値) |
| ゲーム構成 | 全3章 × 各5ステージ(5ステージ目はボス)。ステージ長は曲の長さに連動 |
| 装備 | 頭・体・足・武器の4部位 × 各10種(計40種)。つけ外し自由、裸も可 |
| 敵 | ザコ12種(基本6種+色違い6種) + ボス3種(章ごとに固定) |

## 1. 画面フロー

```
タイトル(タップして開始 = AudioContext解禁)
  └→ モード選択
        ├→ ゲームへ …… 出撃準備画面へ
        ├→ オプション …… 音量(BGM/SE別)・タイミング較正(§2.3)・セーブ初期化
        └→ 曲視聴 …… 解放済みの曲を一覧からフル再生(ジュークボックス)
出撃準備画面
  ├ 章・ステージ選択(クリア済み+次の1つが選択可)
  ├ キャラ選択(ラット/ソードマン/ガンナー ※解放済みのみ)
  ├ 装備(頭・体・足・武器。所持品から選択/外す。効果と説明を表示)
  ├ 曲選択(そのステージで流す曲を選ぶ)
  └ ショップ(コインで装備購入・キャラ解放)
  └→ [いざ!] → ダンジョン入場(曲スタート)
ダンジョン → クリア/ゲームオーバー → リザルト(獲得コイン・最大コンボ・判定内訳) → 出撃準備へ戻る
```

## 2. リズムシステム(最重要・最初に作る)

### 2.1 Conductor(音楽時計)
- **`AudioContext.currentTime` を唯一のマスタークロックにする。**
  `setTimeout` / `performance.now` / rAF のタイムスタンプを拍計算に使うことを禁止する
  (描画補間にのみ rAF を使う)
- 曲データ `{bpm, offset}` から `beatTime(n) = offset + n * 60 / bpm` を計算し、
  「現在の拍位置(小数)」を常時提供する: `currentBeat = (audioCtx.currentTime - startTime - offset - calibration) * bpm / 60`
- BPM変化は譜面データの `bpmChanges` で対応(区間ごとの累積時間テーブルを事前計算)
- 曲は `AudioBufferSourceNode` で再生(HTMLAudio は遅延・シーク精度が不安定なため不使用)。
  ステージ側の都合で曲が終わっていない場合はシームレスにループ再生する

### 2.2 入力判定
- 判定窓(CONFIG集約、初期値): **PERFECT ±60ms / GOOD ±120ms / それ以外 MISS**
- PERFECT/GOOD → 行動成立(PERFECTは威力・移動にボーナス)。MISS → 不発+コンボ切れ
- 同一拍への二重入力は無効。次の拍の窓が開くまで入力はバッファしない(シンプル優先)
- 判定結果は判定点の上に「PERFECT/GOOD/MISS」をポップ表示

### 2.3 レイテンシ較正(初期実装に含める・後回し禁止)
- オプションに「較正モード」: メトロノームに合わせて8回タップ → 平均ズレを `calibration`(ms) として localStorage に保存
- 手動±ms調整スライダーも併設
- スマホのタッチ+Bluetooth等の音声遅延は端末差が大きく、これがないと必ず「ズレてる」と感じる

## 3. アクション仕様(拍ゲート式リアルタイム)

見た目は横スクロールのリアルタイムアクションだが、**内部の座標はタイル単位(1タイル=キャラ1体分)で量子化**する。
行動成立時に目標タイルを決め、0.4拍程度のトゥイーンで滑らかに移動させる。
衝突判定・レベル生成・敵AIが全てタイル基準になり、実装が桁違いに堅くなる。

- **◀ / ▶(移動)**: 前方1タイルへダッシュ。目の前が1段の段差なら自動で駆け上がる/降りる
- **ジャンプ**: 上方向へ約2タイルの放物線(約1拍で頂点)。空中でも拍が来ていれば ◀▶ で1タイル空中制御可
- **落下**: 足場がなければ拍と無関係に自動落下(重力はリアルタイム)。着地するまで移動入力は空中制御扱い
- **攻撃**: キャラ固有(§5)。方向ボタンと同拍押しで方向指定攻撃
- **敵との接触**: 敵のいるタイルに攻撃以外で侵入 or 敵の攻撃拍に同じタイルにいるとダメージ+ノックバック1タイル

### コンボとフィーバー
- コンボ = 判定成立(PERFECT/GOOD)の連続数。MISSと被弾で0に戻る
- **画面右上に「コンボ数」と「総タップ数」を小さく常時表示**する
- コンボ中はコンボ数の桁が上がるごとに表示が派手になる(色・パルス)。
  16コンボ以上で**フィーバー**: 攻撃力1.5倍+コインドロップ2倍+画面縁が光る等の演出。MISSで解除

## 4. ステージ長と曲の連動

1ステージは「1曲が終わるか終わらないか」の長さにする。ステージ生成時に曲から逆算する:

```
総拍数 totalBeats = 曲の長さ(秒) × bpm ÷ 60   (bpmChangesがあれば区間積算)
ゴールまでの距離(タイル) = totalBeats × CONFIG.STAGE.ADVANCE_RATE
```

- `ADVANCE_RATE`(初期値 0.45): フルコンボ時の「1拍あたりの平均前進タイル数」の想定。
  戦闘・寄り道・後退のロスを織り込んだ係数で、プレイテストで調整する
- チャンク数 = ゴール距離 ÷ チャンク幅(16) を丸めて生成
- 曲が終わってもゴール未到達なら曲をループして続行(失敗にはしない)。
  曲が終わる前にゴールすると「曲内クリアボーナス」をリザルトで加算

## 5. プレイアブルキャラクター(3種・データ駆動)

| キャラ | 攻撃 | 特性 |
|---|---|---|
| ラット(初期) | **タックル**: 前方2タイルへ突進。経路上の敵にダメージ(移動兼攻撃) | 機動力型。HP少なめ |
| ソードマン(解放) | **剣**: 前方1タイルを薙ぐ(移動しない)。上方向指定で対空斬り | バランス型。HP中 |
| ガンナー(解放) | **ビーム**: 前方直線を貫通攻撃。発射後1拍はリロードで攻撃不可 | 遠距離型。HP低・威力高 |

キャラ定義は `CONFIG.CHARACTERS` に集約(HP/威力/射程/クールダウン/解放価格)。追加はデータを足すだけにする。

## 6. ダンジョン自動生成(チャンク方式)とカメラ

横視点の自動生成は「部屋と通路」より **手作りチャンクのランダム接続(Spelunky方式)** が堅い。

- 1チャンク = 横16×縦10タイルの手作りテンプレート。左右の接続口の高さ(上/中/下)をタグ付け
- 生成時: `スタートチャンク → (接続口が合うチャンクをシード乱数で接続 ※個数は§4で算出) → ゴールチャンク`
- チャンク内に「敵スポーン候補」「アイテム候補」「コイン候補」マーカーを埋め込み、難易度(章が進むほど高)に応じて抽選配置
- テンプレートは `levelgen.js` に文字列アートで定義(例: `"####  ..E..C.  ####"`)。追加が容易で Sonnet に任せられる
- 落下死は作らない(最下段は床 or 落ちたらダメージ+直前足場に復帰)。リズムゲーで即死落下はストレスが強い
- **カメラ**: マップ全体は画面よりはるかに横に長く、画面(窓)はプレイヤーとその周囲だけを映す追従カメラ。
  進行方向がやや先に見えるよう前方オフセットをかけ、移動トゥイーンに合わせて滑らかにスクロールする

## 7. 敵(ザコ12種 + ボス3種・データ駆動)

敵の行動は全て拍駆動。**予備動作(光る等) → 次の拍で実行**を徹底し「音楽を聴いていれば避けられる」を保証する。
定義は `CONFIG.ENEMIES` に集約し、色違いは基本種のパラメータ違い(行動エンジン共通)として実装する。

### 基本6種
| 敵 | 行動 |
|---|---|
| スライム | 2拍に1回、左右にうろつく。接触ダメージのみ |
| バット | 毎拍プレイヤー方向に1タイル接近(空中) |
| ナイト | 2拍に1回接近。隣接すると予備動作→次の拍に攻撃 |
| ガンナー敵 | 4拍に1回、横直線に弾(毎拍1タイル進む) |
| ゴースト | 2拍ごとに実体化⇔透明を繰り返す。透明中は無敵・すり抜け |
| ボマー | 3拍に1回爆弾を設置。爆弾は2拍後に十字1タイルで爆発 |

### 色違い6種(強化版)
| 敵 | 変更点 |
|---|---|
| レッドスライム | 毎拍移動(速い)・HP+1 |
| ゴールドバット | 倒すとコイン大量。プレイヤーから逃げる方向に移動 |
| ブラックナイト | HP2倍・攻撃2。予備動作は同じ |
| スナイパー(青ガンナー) | 2拍に1回発射・弾速2タイル/拍 |
| 怨霊(赤ゴースト) | 実体化した拍にプレイヤー方向へ1タイル突進 |
| デスボマー(黒ボマー) | 爆発範囲3×3・爆弾2個まで同時設置 |

### ボス3種(章ごとに固定・§10)
ボス戦は専用アリーナチャンクで行う。攻撃パターンは拍駆動の固定ローテーション+HP残量でフェーズ変化。
定義は `js/boss.js` にパターンをデータで持つ。

## 8. 拾得アイテム(ダンジョン内ドロップ)

`CONFIG.ITEMS` に集約:
- ハート(HP1回復) / シールド(被弾1回無効) / ブースト(16拍の間、移動距離2倍) / コイン(メタ通貨)

## 9. 装備(頭・体・足・武器 × 各10種)

## 10. ストーリーと章構成

> 世界の音楽は「音喰らい」に呑まれ、静寂に沈んだ。
> リズムの心臓を持つネズミ・ラットは、奪われた「音のカケラ」を取り戻すため、
> 静まり返った世界の奥へと駆け出した——。

- 全3章 × 各5ステージ。**各章の5ステージ目がボスステージ**
- 章クリアで次章解放。章が進むほど敵の配置密度・色違い率が上がる
- 章ごとの見た目テーマ(背景・タイル色)を変える

| 章 | 舞台 | ボス | ボス戦を含むステージ長 |
|---|---|---|---|
| 1章 | 静寂の森 | オオコウモリ「サイレンサー」 | **曲1.5周分** |
| 2章 | ノイズの機械都市 | 鋼鉄ゴーレム「ビートクラッシャー」 | **曲2周分** |
| 3章 | 音喰らいの城 | 音喰らい「ミュートス」 | **曲3周分** |

### ボスステージの構造
- 前半: 通常ステージと同じ自動生成区間(曲の約0.5周分の長さで生成)
- 後半: 固定のボスアリーナ。ボスが出現し、拍駆動の攻撃パターンを避けつつ攻撃を当てる
- ボスのHPは「残りの曲周回数 × 想定DPS」から逆算した値を CONFIG で調整し、規定の長さで倒しきれる設計にする
- 曲はループでつなぐ。ボス撃破 = クリア(ゴール扱い)
- ボスの攻撃パターン例(サイレンサー): 4拍かけて画面端へ移動 → 2拍予備動作 → 突進(横一列) →
  天井から小コウモリ召喚 → HP50%以下で突進が2連続に(フェーズ2)

## 11. メタ進行(ホーム)

- **ショップ**: §9の装備40種をコインで購入。キャラ解放(ソードマン/ガンナー)もここ
- **記録**: ステージごとにクリア/ハイスコア/最大コンボを保存
- セーブは localStorage(バージョン番号つきJSON。将来の項目追加に備える)

## 12. 楽曲・譜面データ

`fetch` は `file://` で動かないため、**譜面はJSONではなくJSファイル**として持つ。

```js
// songs/songs.js
const SONGS = [{
  id: "song01",
  title: "曲名",
  audio: "songs/song01.mp3",
  bpm: 140,
  offset: 0.52,              // 最初の1拍目までの秒数
  bpmChanges: [],            // [{beat: 128, bpm: 150}] 形式
  chart: [                   // 区間ベースの譜面(Claudeが曲構成に合わせて設計)
    { from: 0,   to: 32,  div: 1, mood: "intro"  },  // div:1 = 4分(毎拍)
    { from: 32,  to: 96,  div: 2, mood: "verse"  },  // div:2 = 8分が混ざる
    { from: 96,  to: 160, div: 2, mood: "chorus", fever: true },
    { from: 160, to: 168, div: 0, mood: "break"  },  // div:0 = 休符(行動不可・敵も停止)
  ],
}];
```

- 基本は「毎拍行動可能」で、譜面は**区間の密度・休符・盛り上がり**を記述する(全拍を列挙しない)
- 曲の追加手順: mp3 を `songs/` に置く → Fable 5 が試聴情報(BPM/offset/曲構成)を元に chart を設計 → Sonnet が組み込み
- 曲の長さ(拍数)からゴール距離・ボスHPを逆算する(§4, §10)

```
│ ♥♥♥ 💰34 [曲名/進行バー]   コンボ:12 タップ:345 │  ← HUD(コンボ・総タップ数は右上に小さく)

```

- ノーツレーンは画面下部中央。左右からノーツが判定点(◆)に向かって流れる
- コンボ中の演出: コンボ数のパルス・色変化、フィーバー中は画面縁の発光+背景の拍同期パルス
- 縦持ちで開いた場合は「横にしてください」表示
- 描画は単一 Canvas。論理解像度 1600×900 固定で devicePixelRatio スケーリング
- キャラ・敵はまず図形+目パーツ程度のプレースホルダ描画で作り、絵は最後に差し替え。装備アイコンは絵文字

## 15. 開発ステップと担当

各ステップ: **Fable 5 が実装指示書を書く → 実装エージェントが実装 → Fable 5 がレビュー → コミット/プッシュ**。
1ステップ = 動くものが1つ増える単位。次ステップは前ステップのレビュー完了後に着手。

## 16. 検証方針(全ステップ共通)

- 変更後 `node --check <file>` で構文確認
- `python3 -m http.server <port>` で配信し、Playwright + `/opt/pw-browsers/chromium` のヘッドレスで
  スマホ横画面ビューポート(844×390, isMobile, hasTouch)を開き、pageerror 0件+変更機能の動作を確認
- リズム精度の検証: `?debug=1` で「拍タイミングと入力時刻の差(ms)」をログ表示するデバッグHUDを Step 1 から用意する

## 17. 注意事項

- 音源をリポジトリに直接コミットする方針(ユーザー判断)。ただし public リポジトリのため、

- `AudioContext` はユーザー操作(タップ)後でないと開始できない(iOS Safari)。タイトルの「タップして開始」で resume する
