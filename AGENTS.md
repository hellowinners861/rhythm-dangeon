# AGENTS.md — Codex向け開発指示書(リズムダンジョン)

このリポジトリは「マッドラットデッド型のリズムアクション × 自動生成ダンジョン」。
スマホ横画面・タッチ専用、**素の HTML/CSS/JS + Canvas のみ**(ライブラリ・ビルド・npm依存の追加は禁止)。

**設計の全容は `DESIGN.md` を必ず先に読むこと。** 仕様の疑問はまず DESIGN.md を参照する。
これまでの開発は Claude(設計=メインセッション+実装エージェント)が担当しており、その運用知識を本書に引き継ぐ。

## 他エージェントとの共存(重要)

- `CLAUDE.md` と `.claude/` ディレクトリ(エージェント定義・ディレクター引き継ぎ書)は **Claude用の設定。消さない・書き換えない**

- 素材の発注仕様は `IMAGE_GUIDE.md`

## 鉄則(壊すと事故る設計不変条件)

1. **拍・タイミングの計算に使う時計は `SFX.ctx.currentTime`(AudioContext)のみ。** `performance.now` / `Date.now` / rAFタイムスタンプは描画補間限定
2. **タイル座標(tx,ty)が真実、描画座標(x,y)は演出。** 衝突・接触・ゴール判定は必ずタイル基準
3. 音声ループは `loopStart=offset, loopEnd=offset+totalBeats*beatSec` で譜面ループと周期一致(ボス戦は曲を複数周する)
4. **生成系の乱数は必ずシードPRNG**(levelgen.js の mulberry32)。`Math.random` は演出のみ
5. 敵・ボスの攻撃は必ず**1拍以上のテレグラフ(予備動作)**を挟む。「音楽を聴いていれば避けられる」の担保
6. 譜面は区間モデル(div: 1=4分 / 2=8分 / 0=休符)。**8分はBPM165以上の曲では使わない**(判定窓と半拍間隔が近づき破綻する)
7. **調整値はすべて `js/config.js` の `CONFIG` に集約。** コンテンツ追加(敵/装備/曲/チャンク)はデータを足す方式
8. コメント・UI文言は日本語

## 検証(コミット前に必須)

1. 構文: `for f in js/*.js songs/songs.js; do node --check "$f"; done`
2. 動作: `python3 -m http.server <port>` でリポジトリを配信し、ヘッドレスブラウザ
   (この環境では Playwright + `/opt/pw-browsers/chromium`。無ければ利用可能なheadless Chromiumで可)で
   **スマホ横画面ビューポート 844×390・isMobile・hasTouch** を開き:
   - **pageerror 0件は絶対条件**
   - 変更した機能を**数値で実測**する(座標・HP・コイン・BPM・拍差ms)。「動いてそう」で通さない
   - 見た目の変更は**スクリーンショットを必ず目で確認**(数値が通っても崩れていることがある)
3. `AudioContext` はタップ後でないと開始しない(iOS対策)。テストは必ず開始タップを挟む
4. `?debug=1` で拍タイミングのデバッグHUD。その他のデバッグ入口:
   `?char=rat|sword|gun` / `?chapter=1|2|3` / `Game._gotoBossStage(ch)` / `Game._teleportToGoal()` /
   `Enemies._debugSpawn(kind,tx,ty)` / `Items._debugSpawnCoin(tx,ty,value)` / `Boss._debugSetHp(n)` /
   拍ちょうどの擬似入力は `Input.dispatch(type, Conductor.beatTime(n))`(nは未来の拍)
5. 初回セーブはオープニング/章ストーリーのパネルがボタンを覆う。テストでは
   `SAVE.data.progress.seenOpening=true; SAVE.data.progress.seenChapterIntro=[true,true,true]; SAVE.save()` でスキップ可
6. チャンクテンプレートを追加・変更したら**攻略可能性のBFS検証**(移動1+ジャンプ2+空中制御1の性能モデル)を必ず実行

## 開発の進め方(前任からの引き継ぎ)

### タスク分解
- **1ステップ = 「動くものが1つ増える」単位**で切り、完成条件を観測可能な動作で定義してから書き始める
- 一番リスクが高い(不確かな)部分を最初に最小構成で作って検証する
- 新規3ファイル超・複数システム横断・検証項目10超になりそうなら2つに割る

### 自己検証の習慣
- 実装後は上記「検証」を全部やる。**自分の実装を信じずに実測する**
- 検証が失敗したら、実装バグと断定する前に**まず自分のテストコードを疑う**。よくある罠:
  - シーン遷移・ロード・トゥイーンの**待ち不足**(タップ直後にevaluateしない)
  - `position:fixed` 要素は `offsetParent===null`(可視判定は hiddenクラス等で)
  - ストーリーパネル等のオーバーレイが下のボタンを覆う(仕様どおりの挙動)
  - APIシグネチャの思い込み(呼ぶ前に定義を読む)
- 音のズレ・手触り・難易度はヘッドレスで検証不能。**ユーザーに実機確認を依頼する項目として報告に明記**する

### 判断基準
- 優先順位: ユーザーの明示的要望 > 目立つバグ > 進行中タスクの完了 > 残タスク(下記)
- **ユーザーに聞く**: ゲームの手触り・見た目の好み・難易度・曲の正確なBPM・後戻りできない仕様分岐
- **自分で決めて報告**: 技術的判断・仕様の穴埋め(埋めた内容は明記)・バランス初期値(実プレイ調整前提とコメントに残す)
- ユーザーは「ゲームで遊ぶ人」の目線で話す。報告は「何ができるようになったか」を先頭に、日本語で

## プロジェクト固有の知識

### 曲の追加手順(mp3が来たら)
1. `songs/songNN.mp3` に配置(**私用音源。一般公開・URL拡散しない**)
2. `tools/analyze-song.js` で解析(使い方はファイル冒頭のコメント)。BPM候補・オフセット・エネルギー推移が出る
3. **`ambiguous: true` が出たら必ずユーザーに正しいBPMを確認**(過去に165の曲を110と誤判定した実績がある)
4. 譜面(chart)を設計: エネルギー推移(rmsProf)からサビ=fever、静かな区間に短い休符(div:0)。区間は4拍境界に丸める
5. `songs/songs.js` に追記。曲名はユーザーに確認

### スプライト(assets/)
- グリーンバックPNGを `js/sprites.js` が読み込み時にクロマキー+**右下の「✦」アーティファクト除去**(ユーザーと除去を約束済み。新画像でも必ず除去)
- 新画像の追加 = `Sprites` の MANIFEST にエントリ追加+描画側の分岐(既存の enemy/boss 分岐を踏襲)。画像が無い種は図形フォールバックを維持
- 「緑背景に緑ボディ」の画像は per-image キーイングパラメータ(`{path, key:{gMin,ratio,spill}}`)で対応(enemy_slime 参照)
- プレイヤーの攻撃ポーズ等、砂埃で不透明boxが破綻する画像は `CONFIG.SPRITE.ANCHORS` に手動アンカーを追加する

### 既知の仕様メモ
- 判定オフセット: song01(アダチレイ・アダチレイ)=165BPM/0.24s、song02(2代目閻魔)=135BPM/0.06s。ズレ報告が来たら±20ms単位で調整
- ボスHP(80/200/400)と「曲1.5/2/3周分」の尺は未調整(ユーザーの実プレイ感想待ち)
- とげの鎧(thorns)はナイトの予約攻撃には反撃しない
- 出撃準備画面は縦390pxが非常に窮屈。要素を足したら必ずスクショで溢れチェック
- ユーザーのスマホはブラウザキャッシュで古い版が表示されがち。「直ってない」報告はまずリロードを案内
- リザルト/ゲームオーバー画面はCanvas描画+矩形ヒットテスト(DOMボタンではない)

### 残タスク(優先度順・着手前にユーザーの新要望を確認)
1. **背景・地形タイルの組み込み**(素材待ち): 視差スクロール背景+シームレスタイル描画。素材仕様は IMAGE_GUIDE.md
2. **残りの敵・ボス・アイテム画像の組み込み**(素材が届き次第、MANIFESTにデータ追加)
3. **バランス調整**: ボスHP・敵密度・判定窓・装備価格(ユーザーの実プレイ感想ベース)
4. **曲の追加**(mp3が来たら上記手順)
5. 細かい改善候補: 曲別ハイスコア表示(saveのrecordsは器だけ実装済み)、章別チャンク追加、演出強化

## Git運用

- 開発ブランチで作業し、機能単位の日本語コミット(「何ができるようになったか」+検証結果の要点)
- ユーザーは **main を正** として見る。作業が固まったら PR を作って main へマージ(これまでの運用: PR #1〜#11)
- force push はしない。`songs/` と `assets/` はバイナリ込みでコミットしてよい(公開はしない前提のリポジトリ)
