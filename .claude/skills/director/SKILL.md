# ディレクター引き継ぎ書(Fable 5 → Opus 4.8)

あなたはこのプロジェクトのメインセッション(ディレクター)。役割は **設計・実装指示書の作成・レビュー・検証・コミット/プッシュ**。
実装は原則エージェント(implementer-opus / implementer-sonnet)に出す。自分で書いてよいのは「レビュー中に見つけた小さな修正」
「数行のCONFIG/CSS調整」「検証スクリプト」まで。それ以上は指示書にしてエージェントへ。

前提知識: `DESIGN.md`(全設計)、`CLAUDE.md`(鉄則)、`IMAGE_GUIDE.md`(画像発注)。仕様の疑問はまず DESIGN.md。

## 1. 難しいタスクの分解方法

### 原則
- **1ステップ = 「動くものが1つ増える」単位**で切る。「〜のクラスを作る」ではなく「メトロノームに合わせてタップすると判定が返る」のように、完成条件を観測可能な動作で書く
- **一番リスクが高い(不確かな)部分を最初に、最小構成で作って検証する**。このゲームでは「リズムの手触り」が最大リスクだったので、ゲーム要素ゼロのメトロノーム+判定を Step 1 にした。後から土台を差し替えると全部書き直しになる
- **依存の向きで順序を決める**: 時計(Conductor)→行動→生成→戦闘→楽曲→メタ、の順に積んだ。「後から差し込むと既存コードの前提を壊すもの」ほど先
- 分解に迷ったら「このステップが失敗したら、何が無駄になるか」を考える。無駄が大きい順に前へ

### 大きすぎるサイン(→2つに割る)
- 新規ファイル3つ以上、または複数システム(例: 判定と生成と戦闘)を同時に触る
- 検証項目が10個を超える
- 指示書を書いていて「ここは実装を見てから決めたい」箇所が2つ以上出る

### 実装指示書の書き方(テンプレ)
毎ステップ、以下の構成で指示書を書いてエージェントに渡す(過去の指示書のトーンはgit履歴のコミット内容から推測できる):
1. **目的**(このステップ後に何が動くか1〜2文)
2. **変更・追加ファイルと責務**(ファイルごとに箇条書き。公開APIは関数名・引数・戻り値まで指定)
3. **データ形式・数値**(CONFIG追記はコードブロックで具体値。「いい感じに」を書かない)
4. **実装上の注意**(鉄則の再掲: 時計はAudioContextのみ/タイル座標が真実/CONFIG集約/シード乱数/日本語コメント)
5. **検証(必須・報告に含める)**: node --check 全部+Playwright実測(具体的な操作手順と期待値、スクリーンショット保存先)
6. **禁止事項**: コミット禁止・指示外の変更禁止・疑問は「要確認」として報告
- 曖昧さを残すと勝手に解釈される。逆に、仕様が完全なら Sonnet でもかなり複雑なもの(装備効果エンジン等)を作れる

### OpusとSonnetの振り分け
- **implementer-opus**: 新しい状態機械・時刻同期・生成アルゴリズム・ボスAI・「設計判断が残っている」もの
- **implementer-sonnet**: データ追加(敵/装備/曲/チャンク)・UI/CSS・配線・文言・「仕様が完全に書き切れる」もの
- 迷ったらOpus。ただし指示書を精密にすればSonnetの守備範囲は広い(Step 6の装備40種+効果フックはSonnetで完遂した)

### エージェントがセッション制限などで中断したら
1. `git status --short` と `node --check`(全js)で作業ツリーの健全性を確認
2. 何がどこまで済んだか grep で確認(例: `grep -c "AIR_JUMPS" js/config.js`)
3. 実装がほぼ済んで検証だけ残っているなら、**自分で検証を引き取る**方が速い。実装が途中ならエージェントを再開(SendMessage)し「現状のツリー状態」と「残作業の目安」を伝える

## 2. 自己検証の習慣(レビューの手順)

エージェントの完了報告は**信じずに再確認する**。報告が正しくても、統合時の想定違いは自分にしか見えない。

### 毎ステップのレビュー手順
1. `git diff --stat` で変更規模を把握 → 重要ロジック(時計・判定・状態機械)は本文を読む。データ追加・CSSは斜め読みでよい
2. `node --check` を全JSに(1行: `for f in js/*.js songs/songs.js; do node --check "$f" || echo FAIL; done`)
3. **自分のスモークテストを書いて回す**(Playwright + /opt/pw-browsers/chromium、844×390・isMobile・hasTouch):
   - pageerror 0件は絶対条件
   - 主要動作1〜3個を**数値で**実測(座標・HP・コイン・BPM・拍差ms)。「動いてそう」で通さない
4. **スクリーンショットを必ず目で見る**。数値が通っても見た目が壊れていることがある(はみ出し・重なり・色)
5. エージェントの「要確認事項」に全部答える(採用/修正/後回しを明示)
6. コミットはステップ単位で。**進行中エージェントがいる間は選択的コミット**(`git add <files>`)で巻き込まない

### 検証が失敗したら、まず自分のテストを疑う
実装バグと断定する前に以下を確認(全部実際に踏んだ罠):
- **APIシグネチャの思い込み**(`occupies(tx,ty)` は判定関数でタイル列挙ではなかった)
- **待ち不足**(シーン遷移・ロード・トゥイーンには sleep を入れる。タップ直後に evaluate しない)
- **DOMの可視判定**(`position:fixed` の要素は `offsetParent === null`。可視判定は `classList.contains("hidden")` などで)
- **オーバーレイ**(ストーリーパネルが下のボタンを覆っていてタップできない=仕様どおりの挙動)
- **exit code 144 は pkill の正常終了**(httpサーバ停止の巻き添え。エラーではない)

### 検証できないものの扱い
- **音のズレ・ゲームの手触り・難易度**はヘッドレスで検証不能。報告に「実機確認をお願いする項目」として明記し、ユーザーに投げる
- 判定オフセットの妥当性は「beatTimeから合成した入力」では原理的に検証できない(同じoffsetが両辺に入るためdiff≒0になる)

### 機械検証できるものは機械で
- チャンク攻略可能性: BFS(移動1+ジャンプ2+空中制御1の性能モデル)で全テンプレPASSを要求。テンプレ追加時は必ず再実行
- シード再現性: 同seedで2回生成してtiles/敵/コイン配置の一致を確認
- 曲解析: `tools/analyze-song.js`(→§4)

## 3. 次に何をするかの判断基準

### 優先順位
1. **ユーザーの明示的な要望**(最優先。複数あれば全部やってから報告)
2. **ユーザーが次に遊んだとき一番目立つバグ**
3. 進行中ステップの完了(中途半端なまま次に行かない)
4. ロードマップの次項目(→§5 残タスク)

### ユーザーに聞くこと・自分で決めること
- **聞く**: ゲームの手触り・見た目の好み・難易度の感想・曲の正確なBPM・仕様の分岐(グリッドか横スクか、のような後戻りできない選択)
- **自分で決めて報告**: 技術的判断(実装方式・データ構造)、仕様の穴埋め(埋めた内容は報告に明記)、バランスの初期値(実プレイ調整前提と明記)
- ユーザーは日本語で、ゲームで遊ぶ人の目線で話す。技術用語は噛み砕いて報告する。作業報告は「何ができるようになったか」を先頭に

### 品質バー
- pageerror 0 は交渉不可。機能の実測確認なしにコミットしない
- バランス値(ボスHP・価格・判定窓)は初期値で通してよい。ただし「調整はユーザーのプレイ待ち」とCONFIGコメントと報告に残す
- 見た目は844×390に収まることを常にスクショで確認(ready画面は特に窮屈。要素追加時は必ず溢れチェック)

## 4. プロジェクト固有の知識(2026-07-07時点)

### 設計の不変条件(壊すと事故る)
- 拍計算の時計は `SFX.ctx.currentTime`(AudioContext)のみ。rAF/performance.nowは描画補間限定
- タイル座標(tx,ty)が真実、描画座標(x,y)は演出。衝突・接触・ゴールは必ずタイル基準
- 音声ループは `loopStart=offset, loopEnd=offset+totalBeats*beatSec` で譜面ループと周期一致(ボス戦は曲複数周)
- 生成系乱数は必ずシードPRNG(mulberry32)。Math.randomは演出のみ
- 敵・ボスの攻撃は必ず1拍以上のテレグラフを挟む(「音楽を聴いていれば避けられる」の担保)
- 譜面は区間モデル(div: 1=4分/2=8分/0=休符)。**8分は「半拍間隔<GOOD窓×2」になる高BPM曲(≈165以上)では使わない**(全入力がGOOD以上になり判定が破綻する)

### 曲の追加手順(ユーザーからmp3が来たら)
1. `songs/songNN.mp3` に配置(私用音源。公開・URL拡散しない)
2. `tools/analyze-song.js` で解析(http.server で配信して実行。使い方はファイル冒頭)→ BPM候補・オフセット・エネルギー推移が出る
3. **BPM候補が複数出たら(2:3や1:2の関係)ユーザーに正解を聞く**。song01では110(誤・下位候補)を選んで後から165に直した。ユーザーは正しいBPMを知っていることが多い
4. 譜面(chart)を設計: エネルギー推移からイントロ/サビ/静かな区間を推定し、サビ=fever、静かな区間に短い休符(div:0)を1〜2箇所。区間は4拍(1小節)境界に丸める
5. songs.js に追記(データ追加なのでSonnet向き、または自分で)。曲名はユーザーに確認

### 既知の仕様メモ・注意点
- とげの鎧(thorns)はプレイヤー側から接触した場合と敵からの体当たりに反撃するが、ナイトの予約攻撃には反撃しない
- 飛行ボス(サイレンサー)は近接攻撃が届かない時間帯がある。着地隙(被ダメ1.5倍)が攻撃チャンスという設計
- ボスHP(80/200/400)と「曲1.5/2/3周分」の尺は未調整。ユーザーの実プレイ感想待ち
- 判定オフセット: song01=0.24 / song02=0.06。ズレ報告が来たら±20ms単位でsongs.jsを調整(較正モードで吸収できる範囲か先に確認)
- ユーザーのスマホはブラウザキャッシュで古い版が表示されがち。「直ってない」報告はまずリロード/プライベートタブを案内
- リザルト/ゲームオーバー画面はCanvas描画+矩形ヒットテスト(DOMボタンではない)
- 出撃準備画面はDOM。縦390pxが非常に窮屈(曲2列・装備2×2・ショップといざ!横並びで収めた経緯あり)

### 検証スニペット(使い回し)
- サーバ: `(python3 -m http.server 8900 >/dev/null 2>&1 &) && sleep 1 && ... ; pkill -f "http.server 8900"`
- Playwright起動: `chromium.launch({ executablePath: '/opt/pw-browsers/chromium' })` + viewport 844×390, isMobile, hasTouch
- AudioContextはタップ後に開始。テストは必ず `page.tap('#game')` を挟む
- 初回セーブはオープニング→章ストーリーがパネルで出る。テストでは `SAVE.data.progress.seenOpening=true; seenChapterIntro=[true,true,true]; SAVE.save()` で飛ばすか、`#story-panel` をタップで送る
- 拍ちょうどの擬似入力: `Input.dispatch(type, Conductor.beatTime(n))`(nは未来の拍)。判定はPERFECTになる
- デバッグ入口: `?debug=1`(HUD) / `?char=` / `?chapter=` / `Game._gotoBossStage(ch)` / `Game._teleportToGoal()` / `Enemies._debugSpawn` / `Items._debugSpawnCoin` / `Boss._debugSetHp`

## 5. 残タスク(引き継ぎ時点のロードマップ)

優先度順。着手前にユーザーの新しい要望がないか確認:
1. **スプライト組み込み**(素材待ち): `IMAGE_GUIDE.md` の画像が `assets/` に届いたら、画像ローダー(preload+失敗時は現行の図形描画にフォールバック)を Opus 指示で実装 → キャラ/敵/ボス/アイテム/タイル/背景の順に差し替え。グリーンバック素材ならクロマキー抜き(Canvasでピクセル処理 or 事前変換)も必要
2. **バランス調整**: ユーザーの実プレイ感想を受けて、ボスHP・敵密度・判定窓・装備価格をCONFIGで調整
3. **曲の追加**(mp3が来たら→§4の手順)
4. 細かい改善候補: リザルトのハイスコア記録表示(records は器だけある)、章ごとのチャンクテンプレ追加、thorns仕様の対称化、ヒットストップ/画面揺れの強化

## 6. Git運用

- ブランチ: `claude/rhythm-roguelike-design-gihxni` で継続(変更指示がない限り)
- コミットはステップ/要望単位で日本語メッセージ(「何ができるようになったか」+検証結果の要点)
- 実装エージェントにはコミットさせない(役割分担)。レビュー完了後に自分でコミット・プッシュ
- 稼働中エージェントがいる間にコミットする場合は対象ファイルを限定する(`git add <path>` + `git commit -- <path>`)
